## 6. 实训心得

### 6.1 工具链的编码陷阱比代码 BUG 更隐蔽

本次测试发现的**唯一一个 BUG**（BUG-001）不是后端代码错误，而是 PowerShell `-Command` 在 Windows GBK 控制台下发送中文 JSON 时产生的字节级字符损坏。通过 SQL Server `sqlcmd` 直查 `EmotionTags` 表逐条对比 Unicode 码点与 UTF-8 原始字节，才确认根因不是 nvarchar 问题，不是 EF Core 映射问题，而是客户端传输层的编码陷阱。这说明**测试工具本身引入的噪声可能比被测系统更隐蔽**——在把现象归因给"后端 BUG"之前，必须用最底层的工具（sqlcmd 查原始字节）做对照实验。

### 6.2 设计选择 vs BUG：判据是读源码，不是看现象

D1.7（`/api/EmotionTags/anime/{id}/wordcloud` 普通用户返回 403）和 AiAgent Admin 403 这两个现象起初被怀疑是 BUG，但读了 `EmotionTagsController.cs` 第 168 行和 `AiAgentController.cs` 第 13 行后，发现都是显式的 `[Authorize(Roles="Admin")]`——是代码刻意限制，不是意外行为。**如果只看"403 = 出错了"的表面现象而不溯源到具体的代码行，就会把 2 个设计选择误归为 2 个 BUG**，这会扭曲测试报告的真实性。

### 6.3 冷启动门槛是用户行为，不是数据量

在测试 F1 推荐用例时，最初以为"≥N 条标签/情绪记录"就是进入 full 模式的充分条件，但实际经验是：收藏再多、标签再多，如果 status 不是 2（看过）且 rating 不到 8，推荐始终走 popular 模式。读了 `TagProfileBuilder.cs` 第 51 行的 `H = {f.Rating>=8 && f.Status==2}` 才确定硬门槛是"用户至少有一部看过且给了高分的番"，而非"数据量大"。**推荐算法的冷启动条件必须还原到代码里的实际判定逻辑，不能靠直觉推断。**

### 6.4 性能测试的分层收敛：本地接口可以并发，AI 接口必须单点

5 个本地接口（Anime/Recommendations/Favorites/EmotionCurves/Tags）响应极快（p95 ≤ 14ms），用 2 并发 PowerShell `Start-Job` 测 wall 是安全的。但 `/api/pcaia/chat-stream` 走的是 DeepSeek 远程 API，单请求 TTFB 207ms / Total 2291ms，如果并发可能触发 API 限流，影响后续所有测试。**性能测试的并发策略必须按接口的后端依赖分层：本地 DB 查询可以并发，第三方 API 必须单请求。**

### 6.5 缺陷数量不应预设，应真实反映

本轮测试的初始设计覆盖了 12 个一级用例 + 3 个性能用例，不预设"一定有几个 BUG"。最终真实结果是：**1 个 BUG**（还是工具链问题，非代码缺陷）。如果为了"显得有成果"而扩大范围或降低严重度门槛、把 4 条设计选择计入 BUG，报告的真实性就会被破坏。**真实反映比看起来"成果丰富"更重要。**
